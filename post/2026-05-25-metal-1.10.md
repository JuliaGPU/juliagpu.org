## FFTs via MPSGraph and AbstractFFTs.jl

On macOS 14+, Metal.jl [now supports](https://github.com/JuliaGPU/Metal.jl/pull/713) FFTs via the AbstractFFTs interface. It's as easy as:

```julia
using Metal
x = MtlArray(randn(ComplexF32, 2048, 2048))
y = fft(x)  # Just works!
```

Plans and other types of FFTs are also supported. Continuing from the example above:

```julia
pinv = plan_ifft(y)
z = pinv * y
z ≈ x # true
```

Using your Metal GPU allows for much faster FFTs compared to using the CPU-based FFTW.jl or even FFTW with AppleAccelerate. The following table shows benchmark results on a 30-core M2 Max:

| Size | CPU (FFTW)  | CPU (FFTW & AppleAccelerate) | GPU (Metal) |
|------|------------|-------------|----|
| 512×512 | 4.2ms | 766.2μs | 173.4μs |
| 1024×1024 | 19.7ms | 3.7ms | 246.3μs |
| 2048×2048 | 99.5ms | 20.8ms | 588.4μs |
| 4096×4096 | 580.1ms | 99.0ms | 2.5ms |

## Other Functionality

- **Device-side dynamic memory allocation:** Metal.jl 1.10 now supports device-side memory allocation. This enables many functions that previously failed to compile to just work. ([PR \#781](https://github.com/JuliaGPU/Metal.jl/pull/781))
- **GPUArrays RNG:** Metal.jl 1.10 now uses the new GPUArrays RNG for host-side RNG. Old MPS random functionality is still available using `Metal.mps_rng()`. ([PR \#686](https://github.com/JuliaGPU/Metal.jl/pull/686))
- **ComplexF32 division:** [PR \#762](https://github.com/JuliaGPU/Metal.jl/pull/762) implements a ComplexF32 division algorithm that does not widen to Float64 like the Base implementation does.
- **New simdgroup intrinsics:** [PR \#741](https://github.com/JuliaGPU/Metal.jl/pull/741) exposes simd shuflle and simd shuffle xor intrinsics, and [PR \#744](https://github.com/JuliaGPU/Metal.jl/pull/744) exposes simd ballot intrinsics (any, all, vote).
- **MTLResidencySet:** MTLResidencySet and related functionality has been wrapped in `MTL` ([PR \#686](https://github.com/JuliaGPU/Metal.jl/pull/686))

## Performance Improvements

- **Improved TTFX:** Metal.jl now has a [real precompilation workload](https://github.com/JuliaGPU/Metal.jl/pull/747). Locally, `using Metal; Metal.ones(10)` went from 10.5 seconds with Metal.jl 1.9.3 down to 1.9 seconds on 1.10.
- **MPSGraph caching for matmul:** Metal.jl [now caches](https://github.com/JuliaGPU/Metal.jl/pull/722) matmul MPSGraphs, making repeat matrix and vector multiplications using MPSGraphs as fast as when using Metal Performance Shaders directly, but without the [upstream correctness issues](https://github.com/JuliaGPU/Metal.jl/issues/381).
- **Faster GPU -> GPU copies for large arrays:** Previously, Metal.jl arrays that used unified memory (`SharedStorage`) would use cpu copies to avoid the overhead of encoding GPU commands. For large (~32MiB+) copies, it's faster to use GPU blit commands. [PR \#716](https://github.com/JuliaGPU/Metal.jl/pull/716) implements this heuristic.

## Noteworthy Bug Fixes

- **Various large array broadcasting fixes:** [PR \#728](https://github.com/JuliaGPU/Metal.jl/pull/728) addresses a few issues related to broadcasting on arrays with dimension sizes greater than 32-bit integer limits.
- **Error on invalid grid-size launches:** This change is technically breaking in the sense that prior to Metal.jl 1.10, launching a kernel with  one of the grid dimensions greater than `typemax(UInt32)` would silently truncate. Attempting to launch kernels with incorrect grid size now errors. Use grid-stride loops for kernels where you expect to use with arrays larger than this limit. ([PR \#751](https://github.com/JuliaGPU/Metal.jl/pull/716))
